# Snappy — 제품 명세 (PRD)

> 친구가 찍어준 내 인생샷을, 민망함 없이 · 원본 화질로 · 완성해서 받아오는 친구 기반 사진 서비스.
> 최종 갱신: 2026-06-14 (현금 마켓 → 크레딧 호혜 + 수거/완성 모델로 피봇)

---

## 1. 한 줄 정체성

**친구가 찍어준 "남이 본 나"를, 카톡 화질 말고 원본으로 받아와 AI로 완성하는 곳.**

핵심 동사 = **수거(retrieval) + 완성(AI)**. 저장(아카이브)도, 전시(피드)도 아니다.

---

## 2. 차별화 — 왜 카톡·인스타·공유앨범이 아닌가

방향(출처)이 다르다. 경쟁 서비스는 "내가 가진 사진"을 다루고, Snappy는 **"남이 가진 내 사진"**을 다룬다.

| | 인스타 | iOS 공유앨범 / 구글포토 | 카톡 | **Snappy** |
|---|---|---|---|---|
| 사진 출처 | 내가 이미 가진 것 | 다 같이 올린 풀 | 내가 가진 것 | **남이 찍어준, 내가 아직 못 가진 것** |
| 방향 | 나 → 대중 (보여주기) | 공동 풀 | 1:1 운반 | **친구 → 나 (받아오기)** |
| 화질 | 압축 | 원본 | **깨짐** | **원본** |
| 고르기 | 내가 | 안 함 | 안 함 | **친구 투표 + AI 추천** |
| 완성(보정) | 수동 | 없음 | 없음 | **내 스타일 AI 보정** |
| 공개 | 공개 무대 | 그룹 | 1:1 | **친구 비공개** |

> 인스타 = "내가 가진 걸 남에게 보여주기". Snappy = "남이 가진 내 사진을 나에게 가져오기". 방향이 정반대.

**경쟁하지 않는 영역(명시)**: 즉시·대량 운반(카톡), 공개 SNS 피드(인스타), 단순 공유 풀(공유앨범). Snappy는 "잘 나온 컷 전용 레인".

---

## 3. 해결하는 페인

> 친구 폰에 잘 나온 내 사진이 있는데 — ① "사진 줘" 하기 민망하고 ② 받아도 카톡 화질 깨지고 ③ 보정은 안 해주고 ④ 흩어져서 못 모음.

| 페인 | Snappy의 해법 |
|---|---|
| 달라고 하기 민망 | **크레딧 시스템** — 개인 부탁이 아니라 구조(찍어주면 보상) |
| 화질 깨짐 | **원본 화질** 수거 (안드↔아이폰 무관) |
| 보정 안 해줌 | **내 스타일 AI 보정** (크레딧으로) |
| 흩어짐·고르기 힘듦 | **친구 투표 + AI 베스트컷 추천** |

---

## 4. 타겟 · 페르소나

**2030 MZ, 인생샷에 진심인 층.** 핵심은 한 친구 그룹 안에 아래 둘이 공존한다는 것:

- **찍사 친구 (공급)** — "사진 잘 찍어주는 친구". *"공짜 노동·보정 셔틀 그만. 찍어주면 크레딧 쌓이고 인정받고, 그걸로 내 사진도 챙긴다."* → **공급 노드 확보가 서비스 성패의 핵심.**
- **인생샷러 (수요)** — *"친구 폰의 잘 나온 내 사진, 원본으로 받아 AI로 더 예쁘게."*

한쪽 동기가 다른 쪽을 끌어온다(양면 네트워크).

---

## 5. 핵심 루프 — "모임 끝 = Snappy 정리" 리추얼

```
[찍사 친구]                          [피사체 친구]
모임/여행에서 찍음
잘 나온 컷 보내기  ──── 워터마크 미리보기 ────▶  받은함에 도착
                                              (AI/투표로) 베스트컷 고르기
                                              크레딧으로 원본 수거
+1 크레딧 적립    ◀──── 소장 시 ────            (옵션) 내 스타일 AI 보정
```

저빈도(이벤트성) 약점은 **"모임 끝나면 Snappy로 정리"** 라는 반복 리추얼 + 일상 **A/B 투표**(고빈도)로 보완.

---

## 6. 크레딧 호혜 경제

현금 제거 → 친구 사이 결제 어색함 + 정산/사업자 규제 동시 회피.

**규칙 (현재 구현)**
- 사진 1장 = **1 크레딧** (가격 설정 없음)
- 내가 찍어준 사진이 소장되면 → **+1 크레딧**
- 신규 가입 → **+5 크레딧** 시드

**크레딧의 "큰 역할" (방향 — 크레딧에 무게 싣기)**
크레딧이 "내 사진 푸는 것"에만 쓰이면 순환만 하고 의미가 약하다. → **AI 보정·화질 업스케일·인화/굿즈** 등 "갖고 싶은 것"의 화폐로 격상한다.
> 친구 많이 찍어줄수록 → 크레딧 → 내 사진 더 예쁘게/원본화질 → 더 보내고 싶어짐.

**워터마크의 역할 재정의**: "보호 장치"가 아니라 **① 소장 전 미리보기 상태 표시 ② 찍어준 사람(@핸들) 크레딧 노출**. (친구 사이 보호는 약하고 "원본 강조"와 모순되므로 보호 목적은 내려놓음)

## 7. 기능 — 현재 / 다음 / 안 함

### ✅ 현재 구현됨
- 친구(양방향) + 받기 설정, @handle 검색
- 사진 보내기(워터마크) → 받은함 → 소장(크레딧 차감) → 원본 수거
- 크레딧 호혜(가입+5 / 소장 시 +1) — 원자적 RPC
- 친구 비공개 **A/B 사진 투표** (1인1표, 결과 막대·승자)
- 신고/반려, 보낸 사진 현황·히스토리
- 접근 감사 로그(D12)

### 🔜 다음 (우선순위)
1. **원탭 AI 보정** (내 스타일) — 크레딧 sink. 크레딧에 무게를 싣는 핵심
2. **AI 화질 업스케일** — "원본 화질" 정체성과 직결
3. **AI 정리 보조** — N장 중 중복·흐림컷 제외, 베스트컷 자동 추천(투표 보완)
4. **AI 스타일 학습** — 쓸수록 내 취향 = 개인화 락인

### ❌ 안 함
- 공개 SNS 피드 (인스타 아류)
- 풀 수동 편집기 (오버스펙 — AI 원탭이 가치/공수 우위)
- 현금 결제·정산·출금 (규제·어색함)
- 즉시·대량 메신저 기능 (카톡 영역)

## 8. 리텐션 · 락인

진짜 해자는 "공유"가 아니라 누적 자산이다 (공유는 베껴도 이건 못 베낌):
1. **누적된 "남이 찍어준 나"** 라이브러리 — 쌓일수록 이탈 비용↑
2. **나를 아는 AI** (스타일 학습) — 개인화 해자
3. 크레딧 잔액 + 친구 그래프
4. "모임 후 정리" 리추얼 습관

## 9. 기술 스택 / 현황

|---|---|
| 프론트 | TanStack Start (React 19 SSR), Tailwind v4, TanStack Query/Router |

**데이터 모델 (요약)**: profiles · photos(batch/status/uploader_hidden) · purchases(session_id) · friendships · wallet_transactions(크레딧 원장) · photo_purchase_sessions · point_charge_sessions · photo_access_logs · polls/poll_options/poll_votes. 사진 구매는 원자적 RPC `purchase_photos_with_points`(크레딧 1장=1).

## 10. 미결 / 로드맵

- **AI 보정 PoC** — 외부 이미지 API 1개 선정(보정 or 업스케일), 비용 방어 위해 크레딧 과금

- **공급 노드(찍사 친구) 확보 전략** — 온보딩에서 공급자 소구 메시지
- **측정 1지표** — "찍사 친구가 있는 그룹"의 소장 전환율
- **카카오 OAuth** — 사업자 인증 후 (이메일 수집 위해)
- **네이티브(Capacitor)** — 스크린샷 차단·푸시, 웹 검증 후

## 부록 — 폐기된 방향 (기록)

- **현금 사진 마켓**: 친구 사이 현금 결제는 사회적으로 독 + C2C 정산 규제. → 크레딧 호혜로 전환.
- **스냅작가 마켓(B)**: 프로는 이미 이메일·인스타·크몽 채널 보유 → 앱 불필요.
- **공개 투표**: 외모평가·초상권 리스크 → 친구 비공개만.
